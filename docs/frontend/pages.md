# pages

One component per route.

## Files

- `LoginPage` — local + SSO login entry.
- `SsoCallbackPage` — handles SSO popup completion.
- `DashboardHomePage` — landing dashboard.
- `TenantsPage` — cloud tenant management. "Add Tenant" opens `TenantImportDialog` (auto-populates from the provider's connection, for AWS and Azure) or a manual entry form (GCP/OCI, and either provider with no connection yet). `TenantImportDialog`'s copy differs per provider (`PROVIDER_IMPORT_COPY`): Azure's picker genuinely picks tenants, while AWS's picks accounts that get grouped as scopes under one auto-created tenant per connection.
- `AuditJobsPage` — list/create discovery jobs. Its `ProgressDialog` polls `GET /discover/jobs/{id}/status` every 2s and shows an overall `LinearProgress` bar ("N of M resource types collected — X resources found", % complete) plus a per-scope breakdown, live while the job is still running. Per-row actions: View Progress, View Topology, Hub & Spoke Validation (→ `ValidationResultsPage`), and (admin) Delete.
- `AuditWorkflowPage` — admin-only job creation flow. "Confirm Hub Selection" is always clickable, even with zero VPCs/VNets checked — a hub-and-spoke topology with no designated hub is a valid real-world state (e.g. a flat/mesh network), not an incomplete form; `hub_ids: []` is accepted by `POST /validate` and just yields no hub-bypass-routing findings.
- `ValidationResultsPage` — NAVIXA Validate results for a single audit job, reachable from `AuditJobsPage`'s per-row "Hub & Spoke Validation" action (not just inline in the creation wizard). Shows existing findings via `getFindings`; if none exist yet, shows the same hub-selection/analysis-mode/run form as `AuditWorkflowPage` (hub IDs prefilled from the job's persisted `hub_selection`, set server-side once `POST /validate/jobs/{id}/run` has run at least once).
- `ConnectionsPage` — admin-only cloud connection/delegated-auth management.
- `ReportsPage` — browses NAVIXA Discover's current cross-job resource inventory (`GET /reports/resources`, `api/reports.ts`), one tab per cloud provider (AWS/Azure/GCP/OCI). Tenant/Subscription filters cascade (picking a tenant repopulates the subscription/account dropdown via `listScopes`; switching provider tabs resets both), plus a Resource Type filter. "Export CSV" streams the same filtered query through `GET /reports/resources/export` (`exportDiscoveredResources`, which downloads via a blob + temporary `<a>` since a plain link can't attach the bearer token). Clicking a row's eye icon opens a detail `Drawer` with the resource's full raw `attributes` JSON.
- `TopologyPage` — renders a `reactflow` network diagram served directly from NAVIXA Graph (`GET /graph/jobs/{id}/topology`, computed live from Postgres - no sync step needed), via the co-located helper `topology/mapGraphToFlow.ts`, which builds ReactFlow **group nodes**: each `Network` is a `type: "group"` container sized to fit its children, with every `PART_OF` child positioned inside it (`parentNode`/`extent: "parent"`) in a grid layout; nodes with no resolvable owning network land in a trailing "Ungrouped resources" container instead of being dropped. `PEERED_WITH` edges connect network containers directly; `Network` nodes with `is_hub=true` render as the hub (highlighted border/fill). Individual resources render via the custom `topology/TopologyNode.tsx` node type, which shows a real icon per resource type/provider (`topology/resourceIcons.tsx`: Azure gets real draw.io Azure SVGs from `src/assets/icons/azure/`, other providers fall back to a semantically-matched MUI icon vocabulary). The diagram supports fullscreen (toolbar toggle, standard `requestFullscreen`/`fullscreenchange` API) and PDF export (toolbar button, `topology/exportTopology.ts` rasterizes the *entire* graph regardless of current pan/zoom via `getNodesBounds`/`getViewportForBounds` + `html-to-image`, embedded into a `jspdf` document). An admin-only "Explain with AI" control generates/fetches the `topology_explanation` NAVIXA InsightAI insight for the job.
